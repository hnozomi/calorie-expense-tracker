コードレビューを行います。

## Input
- `$ARGUMENTS`: ファイルパス（任意）
- `docs/requirements.md`（存在すれば自動で読み込む — 仕様との整合確認に使用）
- 指定なしの場合は `git diff` で未コミットの変更をレビュー

## Output
- レビュー結果（会話内で報告、ファイル出力なし）

## 指示

`$ARGUMENTS` でファイルパスが指定された場合はそのファイルを、指定がなければ `git diff` で未コミットの変更をレビューします。

`docs/requirements.md` が存在すれば読み込み、仕様との整合性も確認してください。

### チェック観点

1. **仕様との整合性**（`docs/requirements.md` がある場合）
   - 受入基準を満たしているか
   - ユーザーストーリーに記載された機能が正しく実装されているか
   - 仕様に記載されていない独自の挙動を追加していないか

2. **ルール準拠**（`.claude/rules/` を参照）
   - コンポーネント規約（配置、命名、Server/Client 分離）
   - コーディングスタイル（アロー関数、type 使用、バレルエクスポート）
   - `@/` パスエイリアスの使用
   - ファイル配置ルール

3. **TypeScript**
   - `any` 型の使用がないか
   - 適切な型定義がされているか

4. **React / Next.js**
   - Server Component と Client Component の適切な分離
   - 不要な `"use client"` がないか
   - 適切なデータフェッチパターン

5. **セキュリティ**
   - 環境変数やシークレットの露出がないか
   - アクセス制御が適切か
   - ユーザー入力のバリデーション

6. **パフォーマンス**
   - 不要な再レンダリングがないか
   - 適切な Suspense 境界
   - 画像の最適化

7. **アクセシビリティ**
   - セマンティック HTML
   - インタラクティブ要素の ARIA 属性

8. **可読性**
   - コードの意図が読み取りやすいか
   - 命名が具体的で適切か
   - 関数やコンポーネントが大きすぎないか（分割すべきか）

9. **保守性**
   - 変更に強い設計になっているか
   - 責務が適切に分離されているか（UI とロジック、コンポーネント間）
   - 不要な依存関係や密結合がないか

10. **ベストプラクティス**
    - 使用しているライブラリ・ツールの推奨パターンに沿っているか
    - React Hook Form, Zod, TanStack Query, Jotai 等の公式ドキュメントに反した使い方をしていないか
    - 非推奨 API や古いパターンを使っていないか

11. **バグリスク**
    - 境界値や null/undefined の未処理がないか
    - 競合状態（race condition）のリスクがないか
    - 依存配列の過不足（useEffect, useMemo, useCallback）
    - エラーハンドリングの漏れがないか

### 出力形式

指摘を以下の3カテゴリに分類して報告:

- **Critical**: 必ず修正が必要（仕様不整合、セキュリティ、バグ、規約違反）
- **Warning**: 修正を推奨（パフォーマンス、ベストプラクティス、保守性）
- **Suggestion**: あると良い改善（可読性、コードスタイル）

各指摘にはファイル名・行番号と修正案を含めてください。
レビュー完了後、`/test` でテスト実装することを案内する。
