テストを実装します。

## Input
- `$ARGUMENTS`: ファイルパスまたはコンポーネント名（任意）
- `docs/requirements.md`（存在すれば自動で読み込む — 仕様の参照元）
- 指定なしの場合はテストが不足しているファイルを検出して提案

## Output
- テストファイル（`src/__tests__/` 配下）

## 指示

`$ARGUMENTS` で指定されたファイルパスまたはコンポーネント名のテストを作成します。指定がなければ、テストが不足しているファイルを検出して提案します。

### 重要: 仕様ベースのテストを書く
- `docs/requirements.md` が存在する場合:
  - **実装コードを見てからテストを書かない。** まず受入基準・ユーザーストーリーを読み、「仕様上こう動くべき」というテストを先に設計する。
  - その後で実装コードを確認し、テストを実行する。
  - 仕様と実装が食い違っている場合は、テストを実装に合わせず、仕様通りのテストを書いて失敗させ、ユーザーに報告する。
- `docs/requirements.md` が存在しない場合:
  - ユーザーに仕様の確認を求める。「この機能はどう動くべきですか？」と質問してからテストを設計する。
  - ユーザーが仕様を口頭で伝えた場合、それを基にテストを書く。

### テストの種類

1. **ユニットテスト** — ユーティリティ関数、カスタム hooks
2. **コンポーネントテスト** — レンダリング、props の反映
3. **インタラクションテスト** — ユーザー操作（クリック、入力、フォーム送信）
4. **エッジケース** — エラー状態、空データ、境界値

### 網羅性の確認
- 対象ファイルの全 export（関数、コンポーネント）にテストがあるか確認する
- 各関数の正常系・異常系・境界値をカバーする
- コンポーネントの全 props パターン（必須、任意、デフォルト値）をテストする
- 条件分岐（if/switch/三項演算子）のすべてのパスを通す
- テスト作成後、カバーされていないケースがないか一覧で報告する

### ルール
- テストファイルは `src/__tests__/` に `[source-name].test.tsx` の命名で作成
- Vitest + Testing Library を使用（既存の `vitest.setup.ts` の設定を活用）
- `toBeInTheDocument()` など Testing Library の慣例的な matcher を使用
- モックは最小限にし、実際の動作に近いテストを書く
- テスト作成後に `pnpm test:run` を実行して全テストが通ることを確認
- 失敗するテストがあれば原因を特定し、仕様起因か実装起因かを判断してユーザーに報告する
