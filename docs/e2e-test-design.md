# E2Eテスト設計書

Playwright による本アプリのE2Eテストの設計・実装状況・カバレッジ分析。
実行方法や罠の要約は `CLAUDE.md` の「E2E Tests」も参照。

## 1. 目的と方針

- **目的**: リリース前に「ユーザーが実際に行う主要フロー」が壊れていないことを機械的に検証する。単体テスト(Vitest)がロジックを守るのに対し、E2Eは画面遷移・フォーム・確認ダイアログ・トーストといった統合された振る舞いを守る。
- **対象**: クリティカルパス(認証 → 食事記録 → 献立 → マスタ管理)と、過去に実バグがあった箇所のリグレッション防止を優先する。
- **非対象**: PWA(Service Worker・オフライン・プッシュ)は devサーバーで再現できないため対象外。OCRは画像フィクスチャが必要なため当面対象外。見た目(ダークモード等)はスクリーンショット比較を導入するまで対象外。

## 2. アーキテクチャ

| 項目 | 内容 |
|------|------|
| ランナー | Playwright(`playwright.config.ts`) |
| ブラウザ | Chromium / Pixel 7 エミュレーション(モバイルファーストのため) |
| 実行 | `pnpm test:e2e`。`NEXT_PUBLIC_E2E=1 pnpm dev` を自動起動(既存サーバーは再利用しない) |
| 並列性 | **直列実行**(`workers: 1`)。全テストが同一ユーザーのデータを変更するため |
| 認証 | `global.setup.ts` がUIログインし `e2e/.auth/user.json`(gitignore)に storageState を保存。各テストはこれを再利用 |
| テストデータ | 専用ユーザー `E2E_EMAIL` / `E2E_PASSWORD`(`.env.local`、gitignore)。**毎回のsetupで全テーブルのデータを削除**してから開始 |
| リトライ | `retries: 1`(devサーバー起因のフレーク対策) |

### データライフサイクル

```
setup: サインイン(supabase-js) → 全データ削除 → UIログイン → storageState保存
テスト: 各テストが必要なデータを自分で作る(前のテストの残骸に依存しない ※例外あり、§5参照)
終了: クリーンアップなし(次回setupで消える)
```

### devサーバー特有の罠と対策(`e2e/helpers.ts`)

| 罠 | 症状 | 対策 |
|----|------|------|
| ハイドレーション前の入力 | DOMには値が見えるがreact-hook-formの内部状態に届かず、送信時に空扱い | `gotoHydrated`(networkidle待ち)+ `fillStable`(値が定着するまでリトライ)+ ログインは fill→submit→URL確認 を一体でリトライ |
| ハイドレーション前のクリック | ドロワー/モーダルを開くクリックが無反応 | `clickToReveal`(結果が見えるまでクリックをリトライ) |
| 開発ツールのオーバーレイ | TanStack Query devtools と Next.js devインジケータ(共に左下)がボトムナビ「ホーム」を遮る | `NEXT_PUBLIC_E2E=1` で両方無効化(`providers.tsx` / `next.config.ts`) |
| 認証レート制限 | 失敗ログインの繰り返しでエラー文言が変わる | 「正しくありません/試行回数が多すぎます」の両方を許容 |
| 曜日依存 | 月曜は「昨日」が前週になり「実績を反映」の対象週が変わる | テスト内で曜日判定して前週へ移動 |

### 対象外(理由付き)

- **PWA/Service Worker**(更新プロンプト、オフライン、通知): devサーバーでSWが本番同等に動作しないため。本番スモークとして手動確認
- **確認メールリンク(サインアップ成功系・コールバック)**: メール受信が必要
- **ダークモードの見た目**: スクリーンショット回帰を導入する場合に検討

## 5. 既知の制約・注意

- **テスト間のデータ依存が一部ある**: 直列実行(spec名のアルファベット順)が前提。既知の依存:
  - M-1が朝食を埋めるため、M-2の `first()` は昼食を指す
  - meal-itemスペックは間食スロットのみを使い、meal-registrationの前提(朝食が空)を壊さない
  - **P-4はMI-2が残した「今日の間食の未転記プランE2Eポテチ」に依存**する(MI-2を変更する場合はP-4も見直すこと)
  - ケース追加時は専用スロット・専用データ名(`E2E`プレフィックス)を使うこと
- **実DB(本番Supabase)を使用**: テストユーザーのデータのみ操作する。他ユーザーへの影響はRLSで遮断されるが、`E2E_EMAIL` を実ユーザーに向けないこと(setupが全データを消す)
- **レート制限**: 短時間に何度も全実行するとSupabase認証のレート制限に当たることがある(A-2は両文言を許容済み。setupのログインが429になった場合は時間を置く)
- **`user_settings` はDELETEのRLSポリシーが無く、setupのデータ削除でリセットできない**(削除が黙って0件になる)。設定系テストは初期値を仮定せず、変更したら自分で復元すること(ST-1/ST-2は対応済み)

## 6. 運用ルール

1. **新機能には対応するE2Eケースを追加する**(このドキュメントの§3と§4を更新)
2. **バグ修正時は、そのバグを再現するケースを追加してから直す**(リグレッション防止)
3. デプロイ前チェックは `pnpm lint` → `pnpm test:run` → `pnpm test:e2e` → `pnpm build`
4. セレクタは `getByRole` / `getByLabel` を優先(アクセシブルな実装を強制する副作用がある)。CSSクラスセレクタは禁止
5. 入力・クリックは必ず `e2e/helpers.ts` のヘルパーを経由する
