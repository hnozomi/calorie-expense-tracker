# パフォーマンス問題の原因と改善の記録

このアプリで発生した「遷移が遅い・もっさりする」問題の原因と対策のまとめ。
同種の問題を再発させない・新機能実装時に同じ轍を踏まないためのリファレンス。

## 大原則(このリポジトリで学んだこと)

1. **「遷移が遅い」と「表示が遅い」を区別して調べる。** タップしてから画面が切り替わるまでに何かを待たせてはいけない。データ取得はスケルトンを出してから裏でやる。
2. **`router.push()` はプリフェッチしない。** リンク遷移は必ず `<Link>` を使う。動的ルート(`[id]`)への遷移は `prefetch` を明示しないと部分プリフェッチ(ローディング境界まで)しかされない。
3. **ミドルウェアの認証往復は遷移をブロックする。** ページが静的シェル(個人データなし・データはRLS保護のクライアントフェッチ)である限り、RSC/プリフェッチ要求の認証チェックはスキップしてよい。ドキュメントロード時のリダイレクトだけ守れば十分。
4. **一覧で取得済みのデータを詳細画面で取り直さない。** TanStack Queryのキャッシュからシードする(`useListCacheSeed`)。
5. **Suspenseクエリの直列化(ウォーターフォール)に注意。** 常時マウントされた子コンポーネントが `useSuspenseQuery` を持つと、親のクエリ完了後に子のクエリが直列で走る。モーダル等は初回オープンまで遅延マウントする。
6. **体感を疑ったらまず計測。** `performance.now()` ベースで pointerdown → pushState → コンテンツ表示のタイムスタンプを取ると、どの区間が遅いかが一発で分かる(このリポジトリではChrome DevTools MCPの `evaluate_script` で計測した)。

## 事例1: タブ切り替えが時々数百ms固まる(2026-07-05)

**症状**: ボトムナビのタブをタップしても画面が変わらない時間が発生。普段は速いのに時々遅い。

**原因**: タップ時のページデータ(RSCペイロード)取得がミドルウェアの `supabase.auth.getUser()`(認証サーバーへのネットワーク往復)を通っていた。実測232〜825ms。Next.jsのルーターキャッシュ(プリフェッチ結果)は**5分で失効**するため、「しばらく使ってから別タブをタップ」すると毎回この待ちが発生していた。

**対策** (`f8b12a3`, `src/proxy.ts`):
`RSC: 1` ヘッダー付きリクエスト(クライアント遷移)でも認証往復をスキップ。実測59〜120msに短縮。

```ts
if (
  request.headers.get("next-router-prefetch") ||
  request.headers.get("purpose") === "prefetch" ||
  request.headers.get("RSC") === "1"
) {
  return NextResponse.next();
}
```

**安全性の根拠**: 全ページが静的シェルで、ユーザーデータはクライアント側でRLS保護のSupabaseクエリとして取得される。認証リダイレクトはドキュメントロード(直接アクセス)時に引き続き実施される。

## 事例2: 計画タブの初回表示が遅い(2026-07-05)

**症状**: 計画タブを初めて開くとスケルトンが長く表示される(実測 約630ms)。

**原因**(2つの複合):
1. `PlanCell` が28セル分の `PlanMenuSelectModal` を常時マウントしており、モーダル内の `useSetMenus()`(Suspenseクエリ)が `meal_plans` 取得(447ms)の**後に直列**で走っていた(185ms)。さらに不要な `recipes` / `food_masters` フェッチも発生。
2. 初回タップ時点でデータキャッシュが空。

**対策** (`f8b12a3`):
- `PlanCell` のモーダルを**初回オープンまで遅延マウント**(`hasOpenedModal` フラグ)。初回オープン時にページ全体がスケルトンに巻き戻らないよう、セル内にローカル `<Suspense fallback={null}>` 境界を設置。
- `TabDataPrefetcher`(`(main)/layout.tsx` に常駐)がアプリ表示直後に「今日の食事・日次サマリー・今週の献立・栄養目標」を `queryClient.prefetchQuery` で事前取得。初回タップでもスケルトンなしで表示される。

## 事例3: 一覧→登録/編集ページの遷移が遅い(2026-07-06)

**症状**: レシピ等の一覧からカードや「+」をタップしても430ms〜1秒画面が変わらない。

**原因**: 遷移がすべて `onClick={() => router.push(...)}` で実装されていた。`router.push` は**プリフェッチが一切効かない**ため、タップ後に動的ルートのサーバーレンダリングを待っていた。

**対策** (`188692e`):
- カード3種(レシピ/食品マスタ/セットメニュー)を `<Link href>` で包む
- ヘッダーの「+」と空状態CTAを `<Link prefetch>` に変更(動的ルートのため完全プリフェッチを明示)

## 事例4: 編集画面到着後にフォームスケルトンが出る(2026-07-06)

**症状**: 遷移は速くなったが、編集画面に着いてからフォームが埋まるまで待たされる。

**原因**(2つ):
1. 詳細クエリ(`use-recipe-detail` 等)が、**一覧で取得済みの完全なデータ**(材料・アイテム込み)をサーバーから取り直していた。
2. カードのLinkがデフォルトプリフェッチ(部分)のままで、タップ時に動的ルートのサーバー往復が残っていた。

**対策** (`e479938`):
- 汎用フック `useListCacheSeed`(`src/hooks/use-list-cache-seed.ts`)を新設。キャッシュ済みの一覧クエリから該当レコードを探し、詳細クエリの `initialData` / `initialDataUpdatedAt` としてシード。一覧から遷移した場合はフォームが即時プリフィルされる(実測36ms・リクエストゼロ)。`initialDataUpdatedAt` に一覧の取得時刻を渡すことで、staleTime超過時は通常どおり裏で再検証される。
- カードのLinkに `prefetch` を明示して完全プリフェッチ化。

**適用条件**: 一覧クエリが詳細画面に必要な全フィールドを返していること(このアプリでは recipes は `recipe_ingredients` 込み、set_menus は `set_menu_items` 込みで取得しているため成立)。

## 事例5: タブ切り替えでスケルトンが毎回チラつく・SWが古いページを配信(2026-07-05, `027318f`)

**症状**: タブを切り替えるたびにスケルトンが一瞬表示される。デプロイ後も古いページが表示されることがある。

**原因と対策**:
- 兄弟Suspenseクエリのウォーターフォール → `usePrefetchQuery` で並列化
- TanStack Queryの `gcTime` が短くタブ切替でキャッシュが消えていた → **30分**に延長(表示は即時、裏で再検証)
- `DailyView` のハイドレーションスケルトンが毎回表示 → 初回マウントのみに限定
- Service WorkerがRSCペイロードをCache-First配信し古いページを固定化 → RSC(`_rsc` クエリ / `RSC` ヘッダー)をSWの対象外にし、Cache-First は immutable な静的アセット(`/_next/static/` 等)のみに限定

## 新しく画面・遷移を追加するときのチェックリスト

- [ ] 遷移は `<Link>` か?(`router.push` はフォーム保存後のリダイレクト等、ユーザーが「押した後に処理を待つ」場面のみ)
- [ ] 動的ルート(`[id]`, `/new`)への `<Link>` に `prefetch` を付けたか?
- [ ] 詳細画面のデータは一覧キャッシュからシードできないか?(`useListCacheSeed`)
- [ ] 常時マウントされるコンポーネントに `useSuspenseQuery` を持ち込んでいないか?(モーダル・ドロワーは遅延マウント)
- [ ] 新しいミドルウェア処理が遷移(RSCリクエスト)をブロックしないか?
- [ ] 体感が悪いときは推測せず、pointerdown → pushState → 表示 のタイムスタンプを計測したか?
