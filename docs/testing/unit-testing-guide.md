# 単体テスト(UT)ガイド

対象: 純粋関数・バリデーションスキーマ・状態ストア・単純なフック。
ツール想定: Vitest(Jestでも同じ)。全体方針は [フロントエンドテスト戦略](./frontend-testing-strategy.md) を参照。

## 1. 対象と責務

| 対象 | テストする観点 |
|------|---------------|
| 純粋関数(utils: 日付、フォーマット、集計、変換) | 同値分割+境界値。入出力のみで完結 |
| バリデーションスキーマ(zod等) | 有効値/無効値/境界/型変換(coerce)の挙動 |
| 状態ストア(Jotai atom / Zustand store) | 初期値、更新、派生値の再計算 |
| 単純なフック(debounce、トグル等) | `renderHook` で入出力。タイマーはfake timers |
| CSV/JSON等の生成 | エスケープ、BOM、空データ、特殊文字 |

**UTの責務は分岐の全網羅**。ここで潰した分岐は、IT/E2Eでは代表1件の確認だけでよい。

**対象外**: 状態機械・mutation・画面ロジックを持つコントローラ系フックはUTではなく**IT**で扱う([ITガイド§5](./integration-testing-guide.md))。UTに置くと依存モックだらけになり、UTの「モックしない」原則が崩れる。

## 2. 設計技法

### 同値分割と境界値分析(基本)

「最小値未満 / 最小値 / 代表値 / 最大値 / 最大値超過」+ 型外(空文字、null)。

```ts
describe("calorieSchema", () => {
  it.each([
    ["0(最小値)は有効", 0, 0],
    ["代表値は有効", 500, 500],
    // 暗黙変換の挙動そのもの(値が0になる)を仕様として固定する
    ["空文字はcoerceで0として受理される", "", 0],
  ])("%s", (_label, input, expectedValue) => {
    const result = calorieSchema.safeParse(input);
    expect(result.success).toBe(true);
    if (result.success) expect(result.data).toBe(expectedValue);
  });

  it.each([
    ["負数", -1],
    ["数値化できない文字列", "abc"],
  ])("%s は拒否される", (_label, input) => {
    expect(calorieSchema.safeParse(input).success).toBe(false);
  });
});
```

有効/無効だけでなく、**parse後の値まで**検証する(coerceのような変換を挟むスキーマは特に。「通るか」と「何になるか」は別の仕様)。

### デシジョンテーブル(条件の組み合わせ)

条件が2つ以上絡む関数(例: 「目標超過なら赤、80%未満なら黄、それ以外は緑」)は、条件×結果の表をそのまま `it.each` に落とす。表がテストコードと一対一になり、レビューで抜けが見える。

### プロパティベース(任意)

「どんな入力でも `parse(format(x)) === x`」のような可逆性・不変条件があるユーティリティには fast-check 等の導入価値がある。通常は境界値で十分。

## 3. フロントエンド頻出の異常系カタログ

テスト設計時にこのリストを毎回なめること。実バグの大半はここから出る。

| カテゴリ | ケース |
|---------|--------|
| 空 | 空文字、空配列、空オブジェクト、0件データ |
| 欠損 | null、undefined、プロパティ欠け |
| 暗黙変換 | `Number("")`→0、`Number(null)`→0、`"0"`と`0` |
| 日付 | 月末+1日、12/31→1/1(年跨ぎ)、うるう年2/29、**UTCとローカルのズレ**(`toISOString()`は日本時間の深夜0-9時に前日を返す) |
| 数値 | 丸め(四捨五入/切り捨ての仕様)、浮動小数点(0.1+0.2)、桁区切り |
| 文字列 | 全角数字、前後空白、絵文字、非常に長い文字列 |
| 一覧 | 1件だけ、同値が複数、ソート安定性 |

## 4. 書き方の規約

- **AAA(Arrange-Act-Assert)** を1テスト1検証で。複数の性質を1つの `it` に詰めない
- **テスト名は仕様文**: 「年を跨ぐ週は両方の年を表示する」のように、テスト一覧が仕様書として読める状態にする
- **`it.each` でパラメタライズ**: 同型のケースは表にする(上記例)
- **時刻は固定する**: `vi.useFakeTimers()` + `vi.setSystemTime()`。「今日」に依存するテストは実行時刻で結果が変わってはならない
- **乱数・UUIDはモック or 非検証**: 値そのものではなく形式や一意性を検証する

## 5. モック方針

**原則モックしない**。モックが必要になった時点でそれはUTの対象ではない(→ITへ)。例外:

- 時刻(fake timers)
- `crypto.randomUUID` 等の非決定値

## 6. 書かなくてよいもの

- TypeScriptの型が保証していること(存在しないプロパティへのアクセス等)
- ライブラリ自体の仕様(zodのcoerce仕様、date-fnsの計算そのもの)。ただし**ライブラリの仕様に依存した自コードの挙動**(空文字がcoerceで0になることを許容するか)は自分の仕様としてテストする
- getter/setterだけの薄いラッパー

## 7. アンチパターン

| アンチパターン | 問題 | 代替 |
|---------------|------|------|
| スナップショットテストの乱用 | 落ちても何が悪いか分からず、無思考に更新される | 意味のある値を明示的にassert |
| 内部関数をexportしてテスト | 実装詳細への固定化 | 公開APIを通して分岐を到達させる。無理なら関数を分割して公開する設計に |
| `toBeTruthy()` の多用 | `1`でも`"a"`でも通る | `toBe(true)` / `toEqual(値)` で厳密に |
| テスト内のif文・try/catch | テスト自体が分岐したら網羅が崩れる | ケースを分ける |
