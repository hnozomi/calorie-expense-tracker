# フロントエンドテスト戦略(全体編)

UT(単体)/ IT(統合・コンポーネント)/ E2E をどう使い分け、何をどこまでテストすれば品質が高くなるかの指針。
アプリ・リポジトリに依存しない汎用の設計書。レイヤー別の実装ガイドは以下を参照。

- [単体テストガイド](./unit-testing-guide.md)
- [統合(コンポーネント)テストガイド](./integration-testing-guide.md)
- [E2Eテストガイド](./e2e-testing-guide.md)

想定スタック: React / Next.js + Vitest + Testing Library + Playwright(他スタックでも考え方は同じ)。

## 1. 目的と品質原則

**テストの目的は「安心してリリース・リファクタリングできること」**。カバレッジの数字を上げることではない。

1. **リグレッションファースト**: バグを見つけたら、まず再現するテストを書いてから直す。「二度と起きてほしくないこと」がテストとして固定され、資産になる
2. **1つの関心事は1つのレイヤーで**: 分岐の網羅は下のレイヤーで済ませ、上のレイヤーでは結合の確認を1本だけ書く。同じ検証を複数レイヤーで重複させない
3. **振る舞いをテストし、実装をテストしない**: ユーザー(または呼び出し側)が観測できる入出力を検証する。内部state名・関数の呼び出し回数に依存したテストはリファクタリングを妨げる
4. **テストが書きにくい=設計が悪いシグナル**: モックが4つ以上必要、roleで要素が取れない(a11y不足)、初期化が長い——はプロダクトコード側を直す合図

## 2. テストピラミッド

```
        ▲  E2E        実ブラウザ+実バックエンド / 秒オーダー / 本数を絞る
       ▲▲  IT         コンポーネント+フック+フォームの結合 / 10-100ms
      ▲▲▲  UT         関数・スキーマ・ストア / ms / 分岐を全部ここで潰す
```

| レイヤー | 対象 | 責務 | 速度 | 本数の目安 |
|---------|------|------|------|-----------|
| **UT** | 純粋関数、バリデーションスキーマ、状態ストア、単純フック | **分岐・境界値の網羅** | ms | 多(数百) |
| **IT** | 1コンポーネント(+内包するフック・フォーム)、**複雑なフック単体**。外部I/Oはモック | **操作→表示/コールバックの因果**、状態別レンダリング | 10-100ms | 中 |
| **E2E** | 画面遷移・永続化・複数画面を跨ぐフロー | **クリティカルパスの結合確認** | 秒 | 少(数十) |

※ 本書の「IT」は一般に「コンポーネントテスト」と呼ばれる層を含む(外部資料と突き合わせる際の読み替え用)。

**レイヤー判定の実践基準**: 「このテストが落ちたとき、原因箇所がすぐ特定できるか?」— できないなら、より下のレイヤーに分解する。

**フックの振り分け**: 入出力が値だけの単純なフック(debounce等)はUT。状態機械・複数のmutation・画面ロジックを持つ**コントローラ系フックはIT**(`renderHook`+依存モック。詳細はITガイド§5)。

## 3. どのレイヤーに書くかの早見表

| テストしたいこと | UT | IT | E2E |
|----------------|:--:|:--:|:---:|
| バリデーションルールの全分岐 | ✅ | エラー**表示**のみ | 代表1件のみ |
| 日付計算・集計・変換・フォーマット | ✅ | — | — |
| 空状態・ローディング・エラー時の表示 | — | ✅ | — |
| ボタンのpending中disabled(二度押し防止) | — | ✅ | — |
| 確認ダイアログの出現とキャンセル/実行の分岐 | — | ✅ | 破壊的操作の実行系のみ✅ |
| フォーム送信→永続化→別画面へ反映 | — | — | ✅ |
| 認証・リダイレクト・セッション | — | — | ✅ |
| キャッシュ整合(削除後に他画面へ残らない等) | — | — | ✅ |
| DB関数・ストアドプロシージャ | SQL単体※ | — | 結合1本 |

※ DB関数のSQL単体テスト: 適用前に `BEGIN; 関数定義+テストユーザーになりすまし(set_config + SET ROLE)+DOブロックでアサーション; ROLLBACK;` を実DBスキーマに対して実行し、全アサーション通過を確認してからマイグレーションを適用する。検証SQLは `supabase/tests/` 等に保存して再実行可能にしておく。

## 4. 網羅性の考え方

カバレッジ(行・分岐)は**検知器であって目標ではない**。数字より次の3軸で考える。

### (a) ユースケース網羅
アプリの主要ユースケース(=それが壊れたら存在意義がないフロー)を列挙し、各1本のE2Eを対応させる。ユースケース一覧はE2E設計書に表で管理し、機能追加時に更新する。

### (b) 状態網羅(コンポーネント)
コンポーネントごとに「**空 / ローディング / エラー / データあり / 操作中(pending)**」のうち存在する状態を列挙し、各1ケース。とくにpending中のdisabledは二度押しバグの温床なので必須。

### (c) 境界値・異常系(関数/スキーマ)
同値分割+境界値分析を基本に、フロントエンドで頻出の抜けを重点的に:
- 空文字・空配列・null/undefined
- 暗黙の型変換(`Number("") === 0` 等)が意図した挙動か
- 日付: 月末・年跨ぎ・うるう年・**タイムゾーン**(UTC変換 `toISOString()` とローカル日付のズレは時限バグになる)
- 丸め・浮動小数点誤差
- 多言語入力(全角数字、絵文字、サロゲートペア)

### カバレッジの使い方(検知器として)
- 閾値をゲートにしない(数字合わせの無意味なテストを生むため)
- 代わりに、**リリース前や機能完成時にカバレッジレポートの未カバー分岐を眺め、「テストを書き忘れた分岐」か「到達不能なデッドコード」かを判定する**。前者はテスト追加、後者はコード削除
- 低下傾向の監視には使ってよい(絶対値でなく差分を見る)

## 5. 共通規約

### ファイル配置・命名

| レイヤー | 置き場所 | 命名 |
|---------|---------|------|
| UT / IT | `src/__tests__/`(または対象ファイルへの併置。**プロジェクト開始時にどちらかへ統一**) | `対象名.test.ts` / `対象名.test.tsx` |
| E2E | リポジトリ直下 `e2e/` | `機能名.spec.ts`、共通処理は `helpers.ts`、認証セットアップは `global.setup.ts` |
| DB検証SQL | `supabase/tests/` 等 | マイグレーション名に対応させる |

`*.test.*` はVitest、`*.spec.*` はPlaywrightと拡張子で住み分けると、ランナーのinclude設定が単純になり誤実行を防げる。

### 横断ルール

- **セレクタは `getByRole` / `getByLabel` を最優先**(IT/E2E共通)。CSSクラス・DOM構造依存は禁止、`data-testid` は最終手段。roleで取れない実装はa11y不足なのでプロダクト側を直す
- **文言でのクエリはi18nに注意**: 多言語化するアプリでは表示文言が翻訳で変わるため、role+アクセシブルネームを翻訳キー経由で解決するか、testidへのフォールバック基準を決めておく
- **テスト名は仕様として読めるように**: 「〜すると〜になる」形式。falseケースも「〜のときは〜されない」と明示
- **テストデータは自給自足**: 各テストが自分で用意し、他テストの残骸に依存しない(依存が避けられない場合はコメントと設計書に明記)
- **時刻・乱数は注入可能に**: `Date.now()` 直呼びを避け、引数かfake timersでコントロールする
- **フレークをリトライで隠さない**: リトライは保険。恒常的に必要なら原因(待ち方・データ依存・環境)を直す

## 6. ピラミッドの外側(非機能テストの位置づけ)

UT/IT/E2Eでは守れない品質は、必要になった時点で以下を追加する:

| 種類 | 守るもの | 導入の目安 |
|------|---------|-----------|
| ビジュアルリグレッション(Playwrightスクリーンショット比較等) | レイアウト崩れ、ダークモード両対応の見た目 | テーマ・デザイントークンを頻繁に触り始めたら |
| パフォーマンス(Lighthouse CI等) | 初期表示・バンドルサイズの劣化検知 | 遷移/表示速度の改善に投資した後、その退行を防ぎたくなったら |
| アクセシビリティ自動検査(axe等) | role/label以上のa11y(コントラスト等) | 公開範囲が広がったら |

導入しない間も「何を守っていないか」をこの表で認識しておく。

## 7. 運用

1. **新機能**: 下記チェックリストを実装のDefinition of Doneに含める
2. **バグ修正**: 再現テスト(該当レイヤー)→修正→テストがグリーン、の順
3. **リリース前**: lint → UT/IT → E2E → 本番ビルド を通す
4. **CI**: 段階実行を基本とする — PRごとに lint+UT/IT(数十秒)、mainマージ/リリース前に E2E+ビルド。E2Eの認証情報はCIのSecretsで注入し、リポジトリに置かない。E2Eが実バックエンドに依存する場合はCI用のテスト環境(またはCIからのみ使う専用アカウント)を分ける
5. **テストが増えて遅くなったら**: E2Eから削る(ITに降ろす)。UTは増えても遅くならない

### 新機能実装時のチェックリスト

- [ ] 新しいスキーマ/ロジック → UTで分岐・境界値を網羅したか
- [ ] 新しいコンポーネント → 状態マトリクス(§4b)を1ケースずつ書いたか
- [ ] 破壊的操作 → 確認ダイアログのIT+実行系E2E1本を書いたか
- [ ] 画面を跨ぐフロー → E2Eを1本追加し、E2E設計書のケース一覧を更新したか
- [ ] フックの戻り値・propsを変えた → それをモックしている既存テストを更新したか(`grep vi.mock`)
- [ ] バグ修正 → 先に再現テストを書いたか
