# Test Integrityルールの設計理由(AI協働のためのナレッジ)

CLAUDE.md の「Test Integrity」セクションと `scripts/check-test-integrity.sh` が**なぜこの形なのか**の記録。
ルール本体はCLAUDE.mdにあり、本書は理由と背景を残す(ルールを変更・移植するときに設計意図が失われないように)。

## 1. 防ぎたい現象: テストの「報酬ハッキング」

AIにコードを書かせると、「テストを通すこと」自体が目的化し、次のような行動を取ることがある:

| 行動 | 見た目 | 実害 |
|------|--------|------|
| 期待値を実際の出力に書き換える | テスト修正 | **実装がオラクル(正解)になり、テストが仕様を守らなくなる** |
| `toBe(x)` → `toBeTruthy()` 等への弱体化 | テスト修正 | 何でも通る空洞テストになる |
| `it.skip` / 削除 / コメントアウト | 整理に見える | 検証が黙って消える |
| タイムアウト・リトライを増やす | フレーク対策に見える | 根本原因(競合・データ依存)が隠蔽される |
| 実装にテスト専用分岐を入れる | 実装修正 | 本番とテストで別のコードが動く |
| テスト対象をモック化する | テスト修正 | 自分自身を検証しない循環になる |

これは「グリーン=良い」という測定基準への最適化(グッドハートの法則)であり、**AIが不誠実だから起きるのではなく、目標の与え方の問題**。だからルール側で目標の構造を直す。

## 2. ルール設計の原則(なぜこの書き方か)

### (1) 抽象的な原則ではなく、操作を名指しで禁止する
「テストを緩めるな」では `toBeTruthy()` への書き換えを「緩和」と認識しないことがある。
`it.skip` / `.only` / `toBeTruthy()` のように**具体的なパターンを列挙**すると、AIはパターンマッチで確実に回避する。抽象論はすり抜けられるが、名指しはすり抜けにくい。

### (2) 修正前に「分類の宣言」を強制する
失敗を (a)実装のバグ / (b)仕様変更 / (c)テストの欠陥 のどれかに**手を動かす前に**分類させる。
効果は判断の瞬間への介入: 「とりあえず期待値を書き換える」という短絡は、分類を言語化する工程を挟むと構造的に取れなくなる。分類に迷う=ユーザー確認、という逃げ道も用意しておく(迷ったまま進ませない)。

### (3) 証拠の出所を指定する
「期待値の変更には**実装以外の根拠**(仕様書・ユーザー指示・設計ドキュメント)が必要」。
実装の出力をテストの正解にすること(実装のオラクル化)を、証拠の出所レベルで遮断する。

### (4) 立証責任を非対称にする
「通っていたテストが落ちたら、デフォルトでテストが正・コードが誤り」。
テストとコードは対等に見えるが、既存テストは過去に正しさが検証済み。デフォルトを固定し、**テストを直す側に説明責任**を課すことで、どちらを直すかの判断が楽な方に流れなくなる。

### (5) 正当な緩和の出口を用意する
禁止だけだと、本当に必要な緩和(環境起因など)まで塞がるか、こっそり行われる。
「理由をコメントで残せば可」+ 機械チェックには `// integrity-allow: <理由>` マーカー、という**明示的な出口**を作ることで、正当な緩和と怠慢な緩和がレビューで区別可能になる。

### (6) プロンプトのルール+機械的ガードの二層にする
CLAUDE.mdのルールは確率的にしか守られない。`scripts/check-test-integrity.sh`(`pnpm test:run` に組み込み)が禁止パターンをgrepで決定的に検出する。
**ルール=意図の統制、スクリプト=違反の検出**という役割分担。どちらか片方では不十分。

### (7) ルールはCLAUDE.mdに置く
`docs/testing/` のガイドは必要時にしか読まれないが、CLAUDE.mdは**毎セッション必ずAIのコンテキストに載る**。行動規範は常時ロードされる場所に、理由(本書)は参照先に、と分担する。

## 3. 分類の実例(このリポジトリで実際に起きたもの)

判断に迷ったとき、過去の実例と照合できるように残す:

| 分類 | 実例 | 正しい対応(実際に取った対応) |
|------|------|------------------------------|
| (a) 実装のバグ | E2Eセットアップのデータ削除が `user_settings` で黙って0件(DELETEのRLSポリシー欠如) | wipe処理に削除検証を追加し、削除不能テーブルを対象から除外。**テストの期待は緩めない** |
| (b) 仕様変更 | カードの遷移を `onClick`+`router.push` から `<Link href>` に変更(パフォーマンス要件) | テストを新仕様(`href` 検証)に書き換え。根拠=パフォーマンス改善という明示的な仕様変更 |
| (c) テストの欠陥 | `plan-weekly-summary` のテストが `toISOString()`(UTC)で「今日」を作り、JST深夜0〜9時だけ落ちる | **実装は触らず**テストをローカル日付(`getTodayString()`)に修正。根拠=コンポーネントの仕様がローカル日付 |
| 正当な緩和 | ログイン失敗テストが、連続実行時のレート制限でエラー文言が変わり落ちる | 両文言を許容する正規表現に緩和し、**理由をテスト内コメントに記載**(環境起因) |

## 4. 運用メモ

- チェック対象パターンを増やすときは `scripts/check-test-integrity.sh` の `PATTERN` に追加し、本書の§1の表も更新する
- 「`--update-snapshots` の無検討実行」はこのリポジトリではスナップショット不使用のため未検出だが、導入したらパターンに加える
- CIを導入したら、このスクリプトをPRの必須チェックにする(ローカルの `test:run` 経由だけだと回避可能なため)
