# E2Eテストガイド

対象: 実ブラウザ+実バックエンドで、画面遷移・永続化・複数画面を跨ぐフローを検証する。
ツール想定: Playwright。全体方針は [フロントエンドテスト戦略](./frontend-testing-strategy.md) を参照。

## 1. 何をE2Eにするか(選定基準)

**入れる**:
- **クリティカルパス**: それが壊れたらアプリの存在意義がないフロー(認証、コア機能のCRUD)
- **境界を跨ぐ因果**: 登録→別画面の集計に反映、A画面の削除→B画面からも消える、双方向の同期
- **破壊的操作の実行系**: 削除・全消去・ログアウト(確認ダイアログ→実行→結果まで1本)
- **過去バグのリグレッション**: 「二度と起きてほしくない」ものを固定する
- **認証境界**: 未認証リダイレクト、ログアウト後に保護ページへ戻れないこと

**入れない**(→下位レイヤーへ):
- 表示バリエーションの網羅(空/エラー/丸め等)→ IT
- バリデーションの分岐 → UT(E2Eは代表1件)
- 環境上再現できないもの(Service Worker、メール受信、OS機能)→ 対象外として設計書に理由を明記

**本数を絞る**: E2Eが遅く・不安定になると開発者が実行しなくなり、スイート全体が死ぬ。1ユースケース1本、全体で数分以内を維持する。

## 2. テスト環境の設計

### バックエンドの選択(最初に決める)

| 選択肢 | 向き・条件 |
|--------|-----------|
| **① テスト専用環境**(別プロジェクト / ローカルSupabase等)| **第一候補**。本番への影響ゼロ、破壊的テストも自由。スキーマをマイグレーションで同期できることが前提 |
| ② 本番環境+専用テストアカウント | 個人開発などで環境を分けるコストが見合わない場合の妥協。**RLSによるユーザー分離が全テーブルで効いていることが条件** |

②を選ぶ場合の防御:
- 削除・書き込みは必ずテストユーザーのIDにスコープする(全件系のクエリを書かない)
- 実ユーザーのメールアドレスを `E2E_EMAIL` に設定しない(セットアップが全データを消す)
- 認証レート制限・負荷を本番ユーザーと共有していることを意識する(連続実行を控える)

### テストユーザーとデータのライフサイクル

- **専用テストアカウント**を用意する(認証情報は `.env.local` 等のgitignore対象に。実ユーザーのアカウントを絶対に使わない)
- **セットアッププロジェクトで毎回リセット**: 実行冒頭でテストユーザーのデータを全削除 → UIログイン → `storageState` を保存 → 各テストはそれを再利用
- **⚠ リセットが効いていることを検証する**: RLSにDELETEポリシーが無いテーブルは、削除が**エラーなしで0件**になる(実際に発生した罠)。wipe処理は削除後の件数確認を入れるか、対象テーブルのDELETEポリシーを確認しておく。リセットできないテーブルに触るテストは初期値を仮定せず、変更したら自分で復元する
- **各テストはデータ自給自足**: 必要なデータは自分で作る。テスト専用の識別子プレフィックス(例: `E2E`)を付け、実データと区別できるようにする
- テスト間のデータ依存が避けられない場合は、**コメントと設計書の両方に依存関係を明記**する(変更時に壊れる場所が分かるように)

### 並列性

同一アカウントのデータを共有するなら**直列実行**(`workers: 1`)。並列化したい場合はテストごとに独立アカウント/テナントを払い出す設計が先。

### 実行環境

- ローカルは dev サーバー起動で十分(Playwright の `webServer` で自動起動)。**テスト用フラグ(例: `NEXT_PUBLIC_E2E=1`)を渡して起動**し、開発オーバーレイ等を無効化できるようにする
- devサーバーを再利用しない設定(`reuseExistingServer: false`)にすると、フラグ無しの手動起動サーバーを掴む事故を防げる
- CIではプロダクションビルド(`build`+`start`)での実行を推奨(ハイドレーション競合が減り、本番相当になる)。認証情報はCIのSecretsで注入する

### ブラウザマトリクス

- 最初は**Chromium+主要ターゲットのデバイスエミュレーション**(モバイルファーストならPixel系)1本で始める(WebKitデバイス定義——iPhone系——を使うにはWebKitバイナリの追加インストールが必要)
- **iOSユーザーが主対象のアプリ(PWA含む)は、安定後にWebKitプロジェクトの追加を検討する**。Safariはドロワー・日付入力・visualViewport等の挙動差が大きく、Chromiumだけでは実機の不具合を検知できない
- 全ブラウザ×全テストにはしない: クリティカルパスの数本だけを第2ブラウザでも回す、で費用対効果を保つ

## 3. フレーク対策カタログ(必読)

E2Eの信頼性は待ち方で決まる。実際に踏みやすい罠:

| 罠 | 症状 | 対策 |
|----|------|------|
| **ハイドレーション競合** | 描画済みでもReactのハンドラ未接続。クリックが無反応、入力がフォームstateに届かず「送信したら空」 | `goto` 後に `waitForLoadState("networkidle")`。それでも競合する操作は「クリック→結果が見えるまでリトライ」「入力→値の定着確認→送信→結果確認を一体でリトライ」(`expect().toPass()`)のヘルパーに集約する |
| **開発ツールのオーバーレイ** | devtoolsボタン・devインジケータが画面隅のUIに重なりタップを横取り | テスト用フラグで無効化 |
| 任意sleep | タイミング依存で不安定 | Playwrightのauto-wait+`expect`のリトライに任せる。`waitForTimeout` 禁止 |
| **日付・曜日依存** | 月曜だけ・月末だけ・深夜だけ落ちる | 「今日」に依存するケースはテスト内で日付を計算して分岐 or 相対操作(「前の日」ボタン)で組む。UTC変換(`toISOString`)を使わない |
| 認証レート制限 | 連続実行でログインが429になり文言が変わる | 失敗系はエラー文言を複数許容。storageState再利用でログイン回数自体を減らす |
| strict mode violation | 同名要素が複数 | `exact: true`、`within(dialog)` でスコープ、UI文言の再考 |
| トーストの取り逃し | 数秒で消える | 表示を`expect`で即座に待つ。トーストに頼らずDB反映後の画面変化も併せて検証 |

**ヘルパーに集約する**: 上記対策は個々のテストに書かず、`e2e/helpers.ts` のような共通関数(`gotoHydrated` / `fillStable` / `clickToReveal` / `loginViaUi`)に閉じ込め、specはヘルパーのみ使う規約にする。

## 4. セレクタとアサーション

- IT と同じ規約: **`getByRole` / `getByLabel` 優先**、CSSセレクタ禁止。roleで取れなければプロダクトのa11yを直す(aria-label追加はテスト容易性とa11yの一石二鳥)
- アサーションは「ユーザーに見える結果」で: URL、見出し、トースト文言、一覧に項目が現れた/消えた
- 副作用の確認は**画面経由**で行う(登録→別画面に遷移して表示を確認)。DBを直接見るのはセットアップ/クリーンアップだけにする

## 5. ケース設計テンプレート

設計書(`e2e-test-design.md` のような文書)に以下の表で管理し、specと1:1対応させる:

| 項目 | 内容 |
|------|------|
| ID | 機能プレフィックス+連番(A-1, P-3 …) |
| 目的 | 何が壊れたら検知したいか(リグレッション対象のバグがあれば明記) |
| 前提 | ログイン状態、依存データ(他テスト依存があれば明記) |
| 手順 | ユーザー操作の列 |
| 期待結果 | 観測可能な結果(URL/表示/文言) |

設計書には併せて「**対象外とその理由**」も書く(SW、メール等)。書かない判断も設計。

## 6. 運用ルール

1. 新しいユースケース追加時は、E2Eを1本追加し設計書のケース一覧を更新する
2. バグ修正時は、可能な限りE2E(またはIT)で再現ケースを先に書く
3. リトライは1回まで。**リトライで恒常的に救われているテストは原因を直す**(§3のカタログを見る)
4. 落ちたらまず trace(`retain-on-failure`)とerror-contextのページスナップショットを見る。「何が表示されていたか」で原因の9割は分かる
5. スイート全体の実行時間を監視し、伸びてきたらITへ降ろせるケースを探す
